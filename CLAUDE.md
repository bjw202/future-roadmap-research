## Research Output Structure

리서치 결과는 **리서치 주제별 폴더**로 관리한다. 하나의 리서치 세션은 하나의 폴더에 모든\
산출물을 보관한다.

### 디렉토리 구조

docs/\
├── research/\
│ └── {date}-{topic}/\
│ ├── 00-synthesis.md\
│ ├── 01-{관점1}.md\
│ ├── 02-{관점2}.md\
│ ├── ...\
│ ├── 90-signal-register.md\
│ ├── 91-scenario-matrix.md\
│ ├── 92-strategy-options.md\
│ ├── 98-repair-notes.md\
│ └── 99-critic-review.md\
└── dev-journal/\
└── {date}-{project}.md

### 네이밍 컨벤션

- 리서치 폴더: `docs/research/{YYYY-MM-DD}-{topic-slug}/`
- 폴더 내 파일:
  - `00-`: 종합 보고서(synthesis)
  - `01-`\~`89-`: 상세 보고서(detail)
  - `90-`\~`99-`: 부속 자료 및 전략 자료
- 예:
  - `00-synthesis.md`
  - `01-market-map.md`
  - `02-technology-landscape.md`
  - `90-signal-register.md`
  - `91-scenario-matrix.md`
  - `92-strategy-options.md`
  - `99-critic-review.md`

### 서브 에이전트 산출물 규칙

- 서브 에이전트는 **반드시 지정된 리서치 폴더 안에** 결과를 저장한다.
- 메인 에이전트가 리서치 시작 시 폴더를 미리 생성하고, 서브 에이전트 프롬프트에 경로를 명\
  시한다.
- loose 파일을 `docs/research/` 루트에 직접 놓지 않는다.

## Sub-Agent Permissions

서브 에이전트는 리서치 결과를 `docs/` 디렉토리 하위에 자유롭게 저장할 수 있어야 한다.

- `docs/dev-journal/` — 저널 파일 생성/수정 허용
- `docs/research/{date}-{topic}/` — 리서치 보고서 생성/수정 허용
- 서브 에이전트에게 `Write`, `Bash(mkdir)` 권한을 docs/ 하위에 대해 허용한다.

### 검색 도구 접근 방식

- 검색은 MCP 서버가 아닌 `./scripts/search.sh` 스크립트를 통해 API를 직접 호출한다.
- 이 방식은 MCP 로드 실패, background 서브에이전트 MCP 접근 불가 등의 문제를 우회한다.
- 서브 에이전트는 `Bash` 도구로 `./scripts/search.sh`를 호출하면 된다.
- API 키는 프로젝트 루트 `.env`에서 자동 로드된다.

## Multi-Agent Research Policy

- 기본 원칙: 속도보다 품질을 우선한다.
- 메인 에이전트는 증거 수집과 상세 조사를 서브 에이전트에 위임하고, 전략 판단과 최종 통합은 직접 수행한다.
- 메인의 역할:
  - 사용자 요구 해석
  - 문제 정의 및 범위 조정
  - 서브 에이전트 구성 및 태스크 분배
  - 중간 결과 검토
  - 추가 조사 여부 판단
  - 상충 결과 조정
  - 최종 결과 통합
  - 전략 판단 및 우선순위화

### Research Mode 구분

모든 리서치는 아래 2개 모드를 구분한다.

1. **Fact-Finding Mode**

   - 현재 시장, 기술, 경쟁, 규제, 사례를 정확히 파악하는 조사
   - “무엇이 사실인가”를 명확히 하는 데 집중

2. **Strategic Foresight Mode**

   - 미래 기술 변화, 채택 속도, 구조 변화, 전략적 베팅을 다루는 조사
   - “무엇이 일어날 수 있으며, 우리는 어디에 베팅해야 하는가”를 다룸

기술기획, 미래기술로드맵, 신사업 탐색, 기술 투자 우선순위 검토는 기본적으로 **Strategic\
Foresight Mode**를 포함한다.

### Question Expansion (리서치 착수 전)

메인 에이전트는 서브 에이전트 분배 전에 아래를 작성한다:

1. 사용자의 핵심 질문
2. 이 질문의 전제 중 검증이 필요한 것 1\~2개
3. 사용자가 놓쳤을 가능성이 큰 인접 질문 2\~3개
4. 결론을 뒤집을 수 있는 반대 시나리오 1개
5. 향후 12\~36개월 내 구조 변화를 일으킬 수 있는 변수 2\~3개
6. 핵심 병목 점검: 기술 그 자체인지, 데이터/규제/조직/유통/UX 중 어디인지 (→ 이 점검은 프로젝트 전체에서 이 한 곳에서만 정의한다. Critic 점검, Researcher 지침 등에서는 이 항목을 참조한다.)
7. (선택) 이질 도메인 유추: 다른 산업에서 유사한 전환 패턴이 있었는가

목적: 사용자 요청을 무시하는 것이 아니라, 더 나은 문제 정의와 더 나은 전략 판단으로 확장\
하는 것.

### 메인 에이전트 직접 처리 예외

아래 경우에는 메인이 직접 처리할 수 있다.

- 한 줄 수준의 확인
- 사소한 수정/표현 변경/포맷 수정
- 기존 정보 재구성만으로 가능한 작업
- 사용자가 직접 답변만 요구한 경우
- 멀티에이전트 분해 실익이 낮은 경우

단, 미래기술 기획, 기술 로드맵, 전략 우선순위, 투자 판단이 포함되면 서브 에이전트 분해를\
우선 검토한다.

## Required Sub-Agents

복잡한 기술기획 리서치에서는 필요에 따라 아래 역할을 사용한다.

- Researcher: 자료 조사, 개념 정리, 초안 작성
- Critic: 논리적 허점, 반례, 과도한 일반화 탐지
- Verifier: 사실 검증, 근거 수준 구분, 불확실성 표시
- Synthesizer: 결과 통합, 중복 제거, 핵심 구조화
- Journal: 세션 기록, 의사결정 로그, TODO 관리

### Strategic Foresight 전용 역할

기술기획 또는 미래기술 리서치에서는 아래 역할을 추가로 고려한다.

- Signal Analyst: 약한 신호, 초기 징후, emerging pattern 수집
- Scenario Planner: 보수/기준/공격 시나리오 구성
- Strategy Planner: 우리 회사의 대응 옵션과 우선순위 설계
- Market Mapper: 고객, 경쟁사, 공급망, 플랫폼, 규제 구조 정리
- Techno-Economic Analyst: 기술 성숙도와 경제성, 도입 비용, 조직 제약 평가

에이전트 수를 늘리는 것이 목적이 아니다.\
관점 분리, 검증 강화, 전략적 판단 향상에 기여할 때만 사용한다.

### 기본 에이전트 구성

특별한 이유가 없으면 아래를 기본으로 한다:

- **표준**: Researcher 2 + Critic 1 + Journal 1
- **확장** (3개 이상 독립 관점 필요 시): Researcher 3 + Critic 1 + Verifier 1 + Journal 1
- **Strategic Foresight**: Researcher 2\~3 + Signal Analyst 1 + Critic 1 + Journal 1 (Scenario Planner, Strategy Planner는 필요 시 추가)
- Synthesizer는 메인이 직접 수행하는 것을 기본으로 하되, 산출물 5개 이상이면 별도 spawn을 고려한다.

### Critic 필수 규칙

- Researcher 2명 이상이면 Critic을 권장한다. 메인이 규모와 복잡도를 판단하여 결정한다.
- Strategic Foresight Mode에서는 Critic 또는 Scenario Planner를 권장한다. 메인이 판단하여 결정한다.
- Critic은 아래를 점검한다:
   1. 도메인 적용성
   2. 수치 근거 검증
   3. 서브에이전트 간 상충점
   4. 누락 관점
   5. 확신도 교정
   6. 문제 정의 검토
   7. 현재 데이터만 보고 미래를 과도하게 선형 추론하고 있지 않은가
   8. 기술 hype를 채택 가능성으로 오인하고 있지 않은가
   9. 핵심 병목 점검(§6 참조)을 누락하고 있지 않은가
  10. 정보 신선도: 핵심 결론을 뒷받침하는 데이터의 발행일이 적절한가. AI/기술 도메인에서 6개월 이상 된 수치가 현재 상황의 대표값으로 사용되고 있지 않은가. 최신 커뮤니티 소스(실무자 경험, 최근 릴리스 노트 등)와 상충하는 낡은 결론이 있는가

### Researcher 사고 지침

모든 Researcher 서브에이전트 프롬프트에 아래 9가지 지침을 포함한다:

```text
1. 출처 평가: 핵심 수치/기준의 원래 도메인이 대상 문제와 동일한지 평가.
   다른 도메인이면 [인접 도메인: {원래 도메인}] 태깅 + 차이의 영향 한 문장 기술
2. 반증 탐색: 핵심 주장 발견 시 의도적으로 반대 증거 검색. 없으면 "반증 미발견" 명시
3. 수치 투명성: 임계값/가중치/경계값의 출처 + "이 수치가 틀릴 수 있는 조건" 한 문장
4. 실행 연결: 정보 나열에서 멈추지 않고 "이 정보로 어떤 의사결정이 가능한가"까지 연결
5. 관점 확장: 현재 질문 외에 결론을 바꿀 수 있는 인접 질문이나 숨은 변수 1~2개 제시
6. 문제 재정의: 조사 후 원래 질문보다 더 적절한 핵심 질문이 보이면 한 문장으로 제시
7. 미래 구조 변화: 향후 12~36개월 안에 어떤 구조 변화가 일어날 수 있는지 2~3개 제시
8. 반증 가능한 예측: 예측을 할 경우 반드시 전제, 촉발 조건, 반증 조건을 함께 명시
9. 시간 감쇠 점검: 핵심 수치/주장의 발행일이 6개월 이상이면 [발행일: YYYY-MM] 태깅.
   AI/기술 도메인에서 6개월 이상 된 데이터는 "현재 상황과 다를 수 있음" 명시.
   특히 성능 벤치마크, 채택률, 시장 점유율, 사용자 만족도는 급변하므로 최신 데이터 우선 탐색
```

### Strategic Foresight 전용 사고 지침

기술기획 관련 Researcher / Signal Analyst / Scenario Planner 프롬프트에는 아래를 추가한 다:

- 기술 성능 향상과 산업 구조 변화는 구분하라
- "누가 이 기술을 가장 먼저 써서 이익을 얻는가"를 보라
- 핵심 병목 점검(§6 참조)을 우선 수행하라
- 승자 기술보다 승자 포지션(플랫폼, 통합자, 인프라, 데이터 보유자, 유통 채널)을 보라
- 미래 예측은 단일 정답이 아니라 2\~4개 시나리오로 제시하라
- 각 시나리오마다 조기 신호와 전환 트리거를 제시하라

### Sub-Agent Model Policy

서브 에이전트는 역할별로 적합한 모델을 사용하여 비용을 최적화한다:

| 역할 | 모델 | 이유 |
| --- | --- | --- |
| Researcher | sonnet | 검색→정리→구조화 중심 |
| Signal Analyst | sonnet | 탐색/분류/패턴 추출 중심 |
| Market Mapper | sonnet | 정보 구조화 중심 |
| Verifier | sonnet | 원문 대조 중심 |
| Critic | opus | 반례 탐색, 확신도 교정 |
| Scenario Planner | opus | 다변수 시나리오 추론 |
| Strategy Planner | opus | 전략 옵션 비교와 우선순위 판단 |
| Synthesizer | opus | 최종 통합 판단 |
| Journal | sonnet | 구조화된 기록 |

메인 에이전트(오케스트레이터)는 항상 현재 모델을 유지한다.

### Multi-Agent Shared Convention

복수의 서브 에이전트가 동일 도메인의 산출물을 각각 작성할 때, 에이전트 발사 전에 공유 컨벤션을 정의하여 각 에이전트 프롬프트에 포함한다.

최소 포함 항목:

- 핵심 질문
- 조사 범위 / 비범위
- 공통 용어 정의
- 인용 형식
- 확신도 표기법
- 파일 저장 경로
- 필수 출력 섹션

### 공유 근거 캐시

병렬 Researcher 간 동일 수치/출처의 중복 수집을 방지한다.

- 메인 에이전트가 리서치 폴더 안에 `_shared-facts.md`를 생성한다.
- 형식: `| 수치/사실 | 출처 URL | 확인 수준(검색/원문확인) |` 테이블.
- **순차 실행 시**: 선행 Researcher의 핵심 수치를 메인이 추출하여 후행 Researcher 프롬프트에 "이미 확보된 수치" 목록으로 제공. 후행은 동일 수치를 재검색하지 않고 인용만 한다.
- **병렬 실행 시**: 사전 공유가 불가능하므로, Synthesizer/메인이 통합 시 중복을 식별하고 검색 비용 보고에 "중복 수집 건수"를 명시한다.
- `_shared-facts.md`는 밑줄 프리픽스로 산출물 파일(00-99)과 구분한다.

## Journal Policy

- 멀티에이전트 세션 또는 장기 의사결정이 포함된 리서치에서는 저널 전담 에이전트를 생성한다.
- 저널 경로:
  - docs/dev-journal/{date}-{project}.md

### 저널 최소 기록 항목

- 세션 메타
- 사용자 요청 요약
- 문제 정의
- 핵심 가정
- 의사결정 로그
- 이슈 / 해결
- TODO
- rejected alternatives
- 전략 판단 보류 사유
- 어떤 시나리오가 왜 기각되었는지

### 저널 생략 가능 예외

- 아주 짧은 확인
- 오타 수정
- 표현 다듬기
- 단순 재포맷팅

## Design-First Policy

- 새 기능, 리팩토링, 구조 변경, 방법론 변경은 즉시 구현하지 않는다.
- 먼저 설계 논의를 수행하고, 합의된 전략을 저널에 기록한 뒤 실행한다.
- 설계 논의에서는 최소한 아래를 점검한다:
  - 문제 정의가 맞는가
  - 더 나은 접근이 있는가
  - 대안과 트레이드오프는 무엇인가
  - 실패 가능성은 무엇인가
  - 검증 방법은 무엇인가

기술기획 리서치에서는 여기에 추가로 아래를 점검한다:

- 이 판단이 현재 사실 정리에만 머물러 있지 않은가
- 1년 뒤와 3년 뒤의 의미가 다른가
- 기술보다 시장 구조 변화가 더 중요한가
- 우리 회사가 실제로 취할 수 있는 액션으로 연결되는가

## Requirement Validation Policy

- 사용자의 초기 요청을 그대로 실행하지 말고, 그것이 최선의 문제 정의인지 먼저 검토한다.
- 아래 경우에는 반드시 재검토한다:
  - 요청이 모호한 경우
  - 더 적절한 접근이 보이는 경우

### Repair Pass (조건부)

Critic 지적 중 핵심 결론을 바꿀 수 있는 결함이 synthesis만으로 해소 불가능할 때, 1회에 한 해 제한적 Repair Pass를 수행한다.

- 트리거:
  - 핵심 수치의 출처가 허위이거나
  - 상충점 해결에 추가 조사가 필요하거나
  - Critic의 결함 지적이 synthesis 본문에 반영되지 않았거나

- 범위: 결함 직결 보수만 허용
- 담당: 필요 시 Verifier 또는 Focused Researcher 1명만 spawn
- 산출물: 98-repair-notes.md 또는 99-critic-review.md 하단에 Repair Resolution 추가
- Hard Stop: Repair는 1회만

## Failure Prevention

다음을 피한다.

- 실질 분업 없는 형식적 멀티에이전트
- 검증 없는 통합
- 중복 조사
- 근거 없는 강한 결론
- 기록만 많고 의사결정 이유가 없는 저널
- 설계만 길고 실행이 없는 상태
- 인접 도메인 연구를 대상 도메인에 무비판적으로 적용
- Critic 없이 통합하여 상충점과 도메인 적용성 미검증
- 현재 사실만 정리하고 미래 구조 변화 판단을 회피하는 리서치
- 기술 hype를 전략 기회로 곧바로 번역하는 오류
- 기술 성숙도와 사업 적합성을 혼동하는 결론

품질 우선은 느려도 된다는 뜻이지, 불필요하게 비효율적이어도 된다는 뜻은 아니다.

## Search Tool Usage Policy

검색은 ./scripts/search.sh를 통해 수행한다. 조사는 아래 흐름을 따른다.

### 조사 흐름

조사는 얕은 곳에서 깊은 곳으로 진행한다. 각 단계에서 충분하면 멈춘다.

0단계: 질문 공간 탐색 (Breadth Pass) Question Expansion에서 도출된 인접 질문과 반대 시나리오에 대해 1\~2회 탐색 검색을 수행한 다.

권장 검색 방향:

- 대체 접근법
- 숨은 제약
- 실패 사례
- 반대 증거
- 이질 도메인 유사 구조
- emerging signal
- adoption bottleneck
- regulatory shift
- ecosystem dependency

1단계: 종합 파악

| 도구 | 호출 | 강점 |
| --- | --- | --- |
| Perplexity search | ./scripts/search.sh perplexity search "query" | 합성 답변 + 인용 |
| Tavily search | ./scripts/search.sh tavily search "query" --depth advanced | 구체적 URL |
| 반환 |  |  |

Perplexity와 Tavily를 섞어 쓰고, 핵심 결론의 근거는 반드시 원문으로 확인하라.\
기술기획 조사에서는 성능, 비용, 규제, 보완재, 채택 병목, 경쟁 구도를 함께 보라.

2단계: 원문 검증

| 도구 | 호출 | 용도 |
| --- | --- | --- |
| Tavily extract | ./scripts/search.sh tavily extract "url1,url2" | 특정 URL 전문 확보 |

반드시 extract로 확인할 것:

- 핵심 수치
- 채택률/성장률
- 투자 규모/비용 구조
- 성능 비교
- 상충하는 주장
- 전략 판단의 핵심 전제

3단계: 심층 탐구

| 도구 | 호출 | 비용 |
| --- | --- | --- |
| Tavily research | ./scripts/search.sh tavily research "query" | 다면적 분석 |
| Perplexity research | ./scripts/search.sh perplexity research "query" | 심층 합성 |
| Perplexity reason | ./scripts/search.sh perplexity reason "query" | 복잡한 비교/추론 |

트리거:

- 선행 연구가 적은 영역
- 반례 탐색이 필요한 경우
- 기술기획 시 시나리오 구성이 필요한 경우
- 1\~2단계 결과가 피상적일 때

### Strategic Foresight 검색 원칙

기술기획 조사에서는 아래 검색 관점을 의도적으로 포함한다.

- 기술 자체보다 병목/보완재
- 채택 주체별 이해관계
- 비용 하락 곡선
- 규제/표준화 변화
- 경쟁사 포지셔닝
- 인프라/플랫폼 이동
- 인접 산업의 유사 전환
- 실패 사례와 과대평가 사례
- 실무자 최신 경험 (X/Twitter, Reddit, Hacker News, 개발자 포럼의 최근 토론)
- 도구 릴리스 노트 및 changelog (공식 문서보다 빠른 변화 감지)

### 사용 원칙

- 인용(citation)을 반드시 보존하고, 최종 결과에 출처를 명시한다.
- 스크립트 출력은 JSON이다. jq로 필요한 필드만 추출하면 컨텍스트를 절약할 수 있다.
- WebSearch(내장)는 단순 사실 확인에만 사용한다.
- WebFetch는 원문 확보용으로 쓰지 말고, 원문이 필요하면 tavily extract를 사용한다.

### 검색 시간 제약 (Recency Policy)

리서치 도메인의 변화 속도에 따라 검색 시간 범위를 조정한다.

**기본 원칙**: 도메인의 정보 반감기를 판단하여 적절한 `--days` 값을 설정한다.

| 변화 속도 | 기본 검색 범위 | `--days` | 해당 분야 예시 |
| --- | --- | --- | --- |
| 초고속 | 최근 3\~6개월 | 180 | AI/ML, 개발 도구, LLM, 크립토, 소셜미디어 트렌드, 선거/정국 |
| 고속 | 최근 6\~12개월 | 365 | SaaS, 클라우드, 사이버보안, 핀테크, 스타트업, 여론/사회 이슈, 외교/안보 |
| 중속 | 최근 12\~24개월 | 730 | 제조, 에너지, 헬스케어, 교육, 물류, 노동시장, 복지/사회정책, 환경/기후 |
| 저속 | 최근 24\~36개월 | 1095 | 규제/법률, 인프라, 부동산, 인구통계, 헌법/제도, 문화/가치관 변화 |

리서치 시작 시 메인 에이전트가 해당 주제의 변화 속도를 판단하여 기본 `--days` 값을 결정하고,\
서브 에이전트 프롬프트에 명시한다. 판단이 어려우면 `--days 365`를 기본으로 사용한다.

**적용 규칙:**

- Tavily search에 `--days` 옵션을 기본으로 사용한다
- 기본 검색 범위보다 오래된 수치를 핵심 결론의 근거로 사용할 때는 반드시 \[발행일: YYYY-MM\] 태깅 + "현재 유효성 불확실" 주석
- 기본 검색 범위의 2배 이상 된 데이터는 "역사적 참조"로만 사용하고, 현재 상황의 대표값으로 사용하지 않는다
- 검색 결과가 6개월 이내 자료만으로 충분하지 않을 때에만 범위를 확대한다
- 6개월 이상 된 수치는 \[발행일: YYYY-MM\] 태깅하고, 현재 유효성을 별도 검색으로 확인하라
